Context System

Authoritative state for location, timezone, and preferences stored in context.json. Never assumed — always checked before creating anything time or location-sensitive.

How it works

Intent verification before every time-sensitive action

  • 📄
    Single source of truth

    Location, timezone, units, and preferences live in one file — context.json. Not inferred from IP address, device settings, or assumptions. Always checked, always authoritative.

  • 🔍
    Intent verification checklist

    Before creating any reminder, cron, or time-sensitive task, the agent runs a mental checklist: Is the timezone correct? Are units right? Are location references accurate? Time of day sensible? All preferences applied?

  • ✈️
    Travel-aware

    When the human travels, location and timezone update in context.json. All subsequent actions (alarms, crons, location-aware alerts) automatically adjust. No separate "travel mode" needed.

  • ⚙️
    Updated explicitly

    The agent never guesses. "I'm in Atlanta now" → agent updates context.json and confirms the timezone. State changes are journaled. Auditable.

See it in action

Intent verification in practice

INC-007 (context assumption incident): Agent assumed timezone from context instead of reading context.json. Prevention task: "Add intent verification checklist to AGENTS.md. Wire context.json check into session-start script." Now every time-sensitive action goes through the checklist.

Intent Verification Checklist

  • Timezone matches the human's current location
  • Units match preferences (miles vs. km, F vs. C)
  • Location references are correct city/region
  • Time of day makes sense in local time
  • All preferences from context.json are applied

Context structure (context.json):

{
  "location": {
    "city": "Raleigh",
    "state": "North Carolina",
    "country": "USA",
    "timezone": "America/New_York",
    "display": "Raleigh, NC"
  },
  "units": {
    "distance": "miles",
    "temperature": "F",
    "volume": "oz"
  },
  "preferences": {
    "briefingTime": "07:00",
    "quietHoursStart": "23:00",
    "quietHoursEnd": "08:00",
    "communicationChannel": "telegram"
  },
  "lastUpdated": "2026-03-01",
  "notes": "Updated for return from travel"
}
