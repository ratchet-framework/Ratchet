Incident Loop

Every failure gets a postmortem. Root cause, blast radius, prevention tasks — all documented before a fix ships. "Done" means the class of problem is addressed, not just the instance.

How it works

Fix the instance. Fix the class. Prevent the class.

  • 📋
    Log it first

    When anything goes wrong — whether the human reports it or the agent notices — create an incident file before touching anything. incidents/INC-NNN-description.md captures what happened, when, and the immediate impact.

  • 🔍
    Root cause, blast radius

    Before fixing anything: what actually caused this? What else could it affect? A surface-level fix without root cause analysis just defers the next occurrence. Map the full blast radius — who or what else is exposed to the same failure mode.

  • ⚙️
    Fix the symptom immediately

    The immediate fix ships in the same session as the incident filing. Don't wait for root cause analysis to be complete — stop the bleeding first, then understand the wound.

  • 🛡️
    Prevention tasks — not just fixes

    Every incident produces at least one prevention task added to the backlog. A prevention task closes the class of problem, not the instance. INC-005 (momentum skipping) and INC-007 (same class) both produced the same prevention: a mechanical gate, not a reminder.

  • ✅
    Close when the class is closed

    An incident isn't resolved when the fix ships — it's resolved when all prevention tasks are complete and verified. Until then, it stays open and surfaces in every briefing.

Log it → Root cause → Fix symptom → Queue prevention → 🔩 Close
The philosophy

A bug caught and logged is a click forward

Finding bugs is good news. Logging them is how we stop repeating them. A bug caught and logged = a click of the ratchet. A bug fixed without logging = a missed opportunity.

Most agents patch symptoms. Ratchet treats every failure as a specimen — extract the class, prevent the class, lock in the improvement. The ratchet only clicks forward because the prevention work actually gets done.

Incident file structure:

incidents/INC-007-context-assumption.md
---
Status: Closed
Severity: P2
Date: 2026-02-28
Root cause: Agent assumed timezone from context instead of reading context.json
Blast radius: Any time-sensitive creation without explicit context check
Prevention tasks:
  - [x] Add intent verification checklist to AGENTS.md
  - [x] Wire context.json check into session-start script
  - [x] Add smoke test: verify timezone in next 3 cron creations
